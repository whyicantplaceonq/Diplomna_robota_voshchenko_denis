# update.md — Оновлення Tycoon Simulator

> Цей документ описує процес оновлення опублікованого острова
> від підготовки до перевірки після виходу нової версії.

---

## 1. Типи оновлень

| Тип | Приклади | Час простою |
|---|---|---|
| **Hotfix** | Виправлення критичного багу | ~15 хв |
| **Minor Update** | Нова покупка, баланс золота | ~30 хв |
| **Major Update** | Нова зона, нова механіка | ~1–2 год |
| **Content Update** | Нові моделі, звуки, мапи | ~1–2 год |

---

## 2. Підготовка до оновлення

### 2.1 Оцінка змін

```
Перед початком оновлення визначте:
  □ Тип оновлення (hotfix / minor / major)
  □ Які файли змінюються (Verse / Content / обидва)
  □ Чи потрібна міграція @editable параметрів
  □ Чи зворотньо сумісні зміни з поточним станом острова
```

### 2.2 Резервна копія поточної версії

```bash
# 1. Переконайтесь що поточна версія закомічена
git status
git log --oneline -5

# 2. Створіть тег для поточної версії перед оновленням
git tag -a v1.0.0 -m "Production version before update"
git push origin v1.0.0

# 3. Збережіть Island Code поточної версії
# UEFN → Island Settings → скопіюйте Island Code
# Запишіть: ISLAND_CODE=XXXX-XXXX-XXXX
```

### 2.3 Перевірка сумісності

```
Verse-зміни:
  □ Нові @editable поля не конфліктують з існуючими
  □ Перейменування класів/функцій не зламає розміщені пристрої
  □ Зміни типів полів не потребують ручного переналаштування

Content-зміни:
  □ Нові ассети проходять ліміт розміру острова (<4 GB)
  □ Нові пристрої сумісні з поточною версією UEFN
```

### 2.4 Планування часу

```
Рекомендований час для оновлення:
  - Робочі дні: 10:00–12:00 або 14:00–16:00
  - Уникайте: вечір/вихідні (пікове навантаження гравців)

Час простою острова під час публікації: 5–30 хвилин
(острів недоступний поки Epic Games обробляє публікацію)
```

## 3. Процес оновлення

### 3.1 Розробка та тестування змін

```bash
# Створіть гілку для змін (Release Flow стратегія)
git checkout main
git pull origin main
git checkout -b release/v1.1

# Внесіть зміни у Verse-код або Content
# ...

# Запустіть статичний аналіз
python scripts/verse_lint.py Verse/

# Скомпілюйте Verse
# UEFN → Verse → Build Verse Code (Ctrl+Shift+B)
```

### 3.2 Локальне тестування

```
UEFN → Launch Session (Ctrl+F8)

Перевірте повний сценарій:
  □ Острів завантажується
  □ Всі нові механіки працюють
  □ Старі механіки не зламані
  □ Немає Verse runtime errors у Output Log
```

### 3.3 Комміт та мерж

```bash
git add Verse/ Content/
git commit -m "feat: <опис змін>"
git tag -a v1.1.0 -m "Release v1.1 — <опис>"

git checkout main
git merge release/v1.1 --no-ff -m "release: merge v1.1 into main"
git push origin main
git push origin v1.1.0
```

### 3.4 Публікація оновлення

```
UEFN → Island → Publish to Fortnite

У діалозі публікації:
  1. Оновіть опис якщо потрібно ("Оновлення v1.1: ...")
  2. "Review" → перевірте метадані
  3. "Publish" → підтвердіть

Очікуйте повідомлення:
  "Your island has been updated successfully!"
```

> **Примітка:** Island Code залишається незмінним після оновлення.
> Гравці автоматично отримають нову версію при наступному вході.

## 4. Оновлення конфігурацій (@editable параметри)

Якщо оновлення додає нові `@editable` поля:

```
1. Відкрийте острів у UEFN після публікації
2. Знайдіть розміщені пристрої unlockables_device
3. В панелі деталей налаштуйте нові поля
4. Збережіть острів (Ctrl+S)
5. Повторно опублікуйте
```

## 5. Перевірка після оновлення

### Обов'язковий чеклист

```
□ Острів доступний за тим самим Island Code
□ Версія відображається в описі острова
□ Всі нові механіки працюють коректно
□ Старі збереження/прогрес гравців не порушені
□ Output Log чистий (немає runtime errors)
□ Продуктивність не погіршилась
```

### Перевірка через Fortnite

```
1. Відкрийте Fortnite
2. Введіть Island Code → запустіть
3. Пройдіть повний сценарій покупок
4. Перевірте що золото списується коректно
5. Перевірте скидання стану при виході
```

## 6. Процедура відкату (Rollback)

Якщо після публікації виявлено критичну проблему:

### Варіант А — Швидкий відкат через Git тег

```bash
# Повертаємось до попередньої стабільної версії
git checkout v1.0.0

# Відкрийте проект у UEFN
# UEFN → Island → Publish to Fortnite
# Публікуємо попередню версію
```

### Варіант Б — Повернення через Git revert

```bash
# Якщо хочемо зберегти історію
git revert HEAD --no-edit
git push origin main

# Відкрийте в UEFN та опублікуйте
```

### Після відкату

```
□ Перевірте що острів повернувся до стабільного стану
□ Задокументуйте причину відкату в GitHub Issues
□ Створіть hotfix гілку для виправлення проблеми
□ Оновіть CHANGELOG.md із зазначенням відкату
```
